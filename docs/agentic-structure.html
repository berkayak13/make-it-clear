Agentic Structure and Runtime Flow

This document explains how the repo currently works as an OpenAI-only Chrome MV3 renarration extension. The important architectural point is that the code is no longer a multi-agent system with separate local workers or competing providers. It is a coordinated pipeline: UI actions send messages to the background service worker, the worker calls OpenAI for extraction and renarration, and content scripts render plain text back into the page.

In this repo, "agentic" means the extension keeps user intent in state, extracts page knowledge, chooses prompts from task, persona, and reading goal context, then runs the next step automatically. The orchestration is in code, not in autonomous agent files.

Mental Model

The current architecture is best understood as a browser-extension pipeline with several agent-like roles. Each role is just a JavaScript module with a clear responsibility.

Coordinator

Background service worker

src/background/main.js initializes defaults and calls setupMessageHandler(). src/background/message-handler.js routes every runtime message to the correct handler.

Extractor

Page knowledge extraction

src/page-flow/extract-page.js combines visible text from the content script with screenshot slices from src/utils/screenshot-capture.js, then asks OpenAI for compact structured JSON.

Writer

Renarration generation

src/page-flow/renarrate-page.js builds the final prompt from extracted knowledge, raw page text, active task, active persona, and saved reading goal. It returns plain text only.

Goal assistant

Chatbot handlers

src/handlers/chatbot.js manages chat sessions, asks OpenAI for replies, extracts a structured reading goal from the transcript, and stores it for later page renarration.

Renderer

Content script UI

content.js owns the selected-text overlay and split panel. Final page renarration is rendered with textContent, which prevents generated HTML from being injected into pages.

OpenAI gateway

Responses API wrapper

src/utils/openai-client.js is the only OpenAI HTTP wrapper. It supports text, image-plus-text, and strict JSON schema calls. Every request includes store: false.

The repo does not currently contain the old local model worker, alternate hosted vision providers, old multi-agent page modules, DOM clone replacement, or old visualizer. Those paths should stay removed.

Runtime Entry Points

manifest.json
Declares the MV3 extension, permissions, popup, side panel, content script, options page, and background service worker path. The background worker points to build/background-entry.js.
vite.config.js
Bundles only src/background/main.js into the build output. Vite also injects VITE_OPENAI_* values from .env.
popup.js
Provides the popup chat UI and the Renarrate Page button. It starts chat sessions, applies reading goals, and triggers the full page renarration message.
sidepanel.js
Provides a larger chat UI and an extraction panel. It can run extraction only, without rendering the final page rewrite.
content.js
Runs on matched web pages. It extracts visible page text, creates the selected-text overlay, opens the split panel, updates progress, and renders the final text.

Full Page Renarration Flow

This is the main "agentic" path: the user asks for a whole page rewrite, and the background worker drives extraction, OpenAI reasoning, storage, and UI rendering.

1. User action

Popup sends a request

The popup button sends run-page-renarration with tabId, page URL, and page title.

2. Orchestration

Background starts one run

orchestrator.js blocks overlapping runs with pageRunInProgress, opens the split panel, and sends progress text to the content script.

3. Extraction

Text and screenshots

extractPageKnowledge() reads visible text from content.js, captures full-page slices, selects up to eight representative images, and calls OpenAI for JSON.

4. Renarration

Prompt with intent

renarratePage() builds a system prompt from task, persona, boilerplate, and reading goal, then sends page knowledge and raw text as the user input.

5. Render

Plain text output

The result is stored in lastPageRenarration and sent to content.js, which writes it into the split panel with textContent.

Popup
popup.js queries the active tab and sends run-page-renarration. The message includes pageMetadata so extraction and storage can preserve URL and title.
Message router
message-handler.js finds the action in a merged handler map: simpleHandlers, chatbotHandlers, and pageFlowHandlers.
Orchestrator
pageFlowHandlers['run-page-renarration'] resolves the active tab, generates a run id, tells the content script to show the panel, and starts extraction.
Content script
extract-visible-page-text walks visible text nodes, skips script/style/form/media tags, ignores the extension UI, filters hidden elements, deduplicates text chunks, and returns title and URL.
Screenshot utility
captureFullPageSlices() scrolls the page, waits for layout to settle, captures visible tab PNG data URLs, retries Chrome capture rate-limit errors, and restores the original scroll position.
OpenAI JSON call
callOpenAIJson() sends the extraction prompt plus selected screenshots to the Responses API. The output must match the strict schema with title, topic, summary, facts, entities, and keyTerms.
Prompt builder
buildRenarrationPrompt() loads current tasks, personas, prompt template, system boilerplate, and chrome.storage.sync.readingGoal. The final system prompt combines all of them.
OpenAI text call
callOpenAIText() asks for final plain text. The prompt explicitly says to preserve facts, avoid invented content, and return readable plain text without HTML or Markdown tables.
Split panel
render-renarration-text reaches content.js. It clears the loading state, creates a div.renarration-final-text, and assigns the model output through pre.textContent.

Chat and Reading Goal Flow

The chat is not the final page writer. Its job is to capture what the user wants from reading, convert that conversation into structured preferences, and save those preferences for later renarration.

Conversation loop

  • chatbot-new-session creates a session with a stable user id and empty messages.
  • chatbot-send appends the user's message, loads recent preference history, and calls OpenAI.
  • The chatbot system prompt asks one question at a time and formats quick replies with >>.
  • The popup and side panel parse >> lines into clickable quick-reply buttons.

Goal extraction

  • chatbot-set-reading-goal builds a transcript from session messages.
  • It loads src/prompts/goal-extraction.md.
  • OpenAI returns JSON with reading goal, depth, focus areas, output style, notes, and language.
  • The UI previews the goal. Only after the user applies it does it become chrome.storage.sync.readingGoal.

How the goal affects the page writer

During final page renarration, renarrate-page.js receives the extraction result and asks buildRenarrationPrompt() for the active prompt. That prompt includes the saved reading goal. Then the user message sent to OpenAI repeats the saved goal before page knowledge and visible page text. This means the model sees the user's intent in both the system-level instruction and the page-specific input.

Selected Text Flow

Selected text renarration is the smaller path. It does not capture screenshots and does not run page knowledge extraction. It sends only the selected text plus the active task/persona prompt context.

Content script
On mouseup, content.js checks the current selection. If the selected text is longer than ten characters, it shows a small R trigger beside the selection.
Background handler
Clicking the trigger sends renarrate-text. src/handlers/simple.js calls renarrateText(), which builds the current prompt and calls callLLM().
Overlay
The response returns to the content script and appears in the floating overlay. Feedback buttons can send submit-feedback events to research storage.

Message Contracts

Runtime behavior is organized around Chrome extension messages. These are the important actions to know.

run-page-renarration
Full page flow. Opens the split panel, extracts knowledge, generates final text, stores the result, and renders it on the active page.
extract-page-knowledge
Extraction-only flow used by the side panel. Produces structured page knowledge and stores lastExtraction.
get-last-extraction
Reads the most recent extraction from local storage so the side panel can restore state.
extract-visible-page-text
Content script action that returns visible, deduplicated page text, title, and URL.
show-renarration-panel
Content script action that creates the split panel and loading state.
update-renarration-progress
Content script action for status text and error text while a page run is in progress.
render-renarration-text
Content script action that renders the final page output using textContent.
renarrate-text
Selected-text flow. Sends selected text through the active prompt and shows the result in the overlay.
chatbot-*
Chat session, chat response, persona extraction, persona apply, and reading-goal extraction actions.
submit-feedback
Stores thumbs-up, thumbs-down, or correction feedback for a renarration run.

OpenAI Request Layer

src/utils/openai-client.js centralizes OpenAI calls. It reads build-time config from import.meta.env and sends requests to https://api.openai.com/v1/responses.

callOpenAIText()

Sends a system prompt as instructions, text input as input, and requests plain text output. Used for chat, selected text renarration, and full page renarration.

callOpenAIJson()

Builds a multimodal content array with input_text and optional input_image parts. It requests strict JSON schema output for extraction.

createResponse()

Adds store: false, handles the API key requirement, enforces timeout behavior, parses API errors, and returns the raw Responses API JSON. 

VITE_OPENAI_API_KEY       required
VITE_OPENAI_TEXT_MODEL    defaults to gpt-5.5
VITE_OPENAI_VISION_MODEL  defaults to text model
VITE_OPENAI_IMAGE_DETAIL  defaults to high
VITE_OPENAI_TIMEOUT_MS    defaults to 120000

Failure Handling

Run-level guard

pageRunInProgress prevents two full page renarrations from running at the same time. The flag is reset in a finally block, so failed runs do not permanently lock the pipeline.

Tab and extraction errors

If there is no active tab, no extractable text/screenshots, or a Chrome API failure, the handler returns { success: false, error }. Full page errors are also shown in the split panel.

OpenAI errors

Missing API keys, timeout, non-OK API responses, empty output, and JSON parse failures become thrown errors in openai-client.js and are returned through the message handler.

Best-effort UI updates

Messages to tabs and status broadcasts are wrapped so panel update failures do not crash unrelated work. Research logging is also best-effort.

Current Boundaries

active

What exists now

  • OpenAI Responses API wrapper for text, vision, and strict JSON output.
  • Page extraction from visible text plus screenshot slices.
  • Final plain-text page renarration in a split panel.
  • Selected-text overlay renarration.
  • Chat-based reading goal extraction.
  • Options-managed tasks, personas, and prompt template.
do not reintroduce

Removed architecture

  • Local model worker code.
  • Alternate hosted vision providers.
  • Old multi-agent page modules.
  • DOM clone replacement.
  • Old visualizer path.

The most important security and UX rule in the current flow is that generated page renarration remains plain text. content.js should keep final page output on textContent, not generated HTML.
