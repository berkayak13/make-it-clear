Side Panel · Reading Assistant

How the Chat Works

The side panel chat is a goal-oriented assistant. The user talks to it about what they want from web content; once the conversation has enough signal, a separate extraction pass turns the transcript into a structured readingGoal that the renarration pipeline consumes.

This document traces a single message round-trip — UI → background handler → OpenAI Responses API → persistence → render — and then covers the goal-extraction branch, storage layout, and a few gotchas worth knowing.

1 · UI surface

What the side panel renders

Everything lives in sidepanel.html and is wired up by sidepanel.js. There is no framework — just DOM nodes with stable IDs that the script grabs on DOMContentLoaded.

Chat surface

#chatMessages holds a stack of bubbles, each tagged with a role class (user or model). A system variant is used for inline status notes (e.g. "Reading goal applied"). The input row holds a textarea with auto-resize and a send button.

Action buttons

The Set Reading Goal button is hidden until the user has sent at least two messages (userMessageCount ≥ 2). The + in the header starts a brand-new session and clears the transcript.

Goal preview

After extraction succeeds, #goalPreview shows the parsed JSON fields and offers Apply Goal (writes to chrome.storage.sync.readingGoal) or Discard.

Extraction panel

A separate #extractionPanel sits above the chat and is owned by setupExtractionPanel(). It triggers the page-knowledge extractor and listens for extraction-progress / extraction-update broadcast messages from the background. It is independent of the chat — mentioned here only so its wiring is not mistaken for chat logic.

Refinement banner

On open, the panel sends check-feedback-trends to the background. If the response carries shouldRefine: true, a banner suggests refining preferences. Clicking it pre-fills the input with a prompt and focuses the textarea — it does not send automatically.

Session restore. On load the panel reads currentChatSessionId from chrome.storage.local and asks the background for the saved messages via chatbot-get-session. If the lookup fails, it falls through to startNewSession().

2 · Message round-trip

What happens when the user hits send

1 · UI

sendMessage() appends a user bubble, increments the counter, and posts chatbot-send to the background with the session id and message text. A "Thinking..." typing bubble is shown.

2 · Handler

chatbotHandlers['chatbot-send'] loads the session from IndexedDB, pushes the user turn into session.messages, and re-persists. Roles in storage are user / model.

3 · Prompt build

Loads chatbot-system.md (cached), substitutes {preferences} with the last 5 stored userPreferences rows, and maps each session message to { role: user|assistant, content } before dispatch.

4 · OpenAI

callLLM() flattens the messages into a single userText blob and calls the Responses API via callOpenAIText(). Model defaults to VITE_OPENAI_TEXT_MODEL. store: false is forced.

5 · Render

The reply is persisted as a model turn, a redacted log row goes into researchLogs, and the text returns to the UI. Lines beginning with >>  become quick-reply chips; the rest becomes a model bubble.

Detailed sequence

sidepanel.js

chrome.runtime.sendMessage({ action: 'chatbot-send', sessionId, message })

Blocks the send button, hides quick replies, removes the welcome card on the first turn.

src/handlers/chatbot.js

session = await researchGet('chatSessions', sessionId) · push user turn · researchPut(...).

prefSummary = getRecentPrefSummary(userId) — last 5 entries from the userPreferences store, sorted by timestamp.

systemPrompt = template.replace('{preferences}', prefSummary).

Re-maps session roles for the LLM: { user: 'user', model: 'assistant' }.

src/utils/llm-dispatch.js

Concatenates the message list into a single multiline string of the form role: content and forwards to callOpenAIText.

Wraps the call in a try/catch and returns { success, result, response } — errors never throw past this layer.

src/utils/openai-client.js

POST https://api.openai.com/v1/responses with instructions: systemPrompt, input: userText, text.format.type: 'text', and an AbortController guarding the configured timeout (default 120s).

getResponseText() prefers output_text, falling back to walking output[*].content[*].text.

src/handlers/chatbot.js

Pushes the model turn into session.messages, persists again, writes a truncated entry (first 500 chars of each side) to researchLogs when enableResearchLogging is not explicitly false.

Returns { success: true, reply } to the side panel.

sidepanel.js

renderModelReply(text) splits on newlines, partitions >> option lines into quick-reply chips, and renders the rest as a model bubble. The send button re-enables and the textarea is refocused.

The full transcript is re-sent every turn. There is no streaming and no server-side conversation state — the session lives in IndexedDB and is replayed into the model on each message. Long sessions grow the token bill linearly.

3 · System prompt & quick replies

The conversation contract

src/prompts/chatbot-system.md is the contract between the UI and the model. Two rules it enforces matter for the rendering pipeline:

  • One question, 3–4 options, each prefixed with >>  — the UI splits those lines out as clickable chips. Free-form text is always allowed too; the model is told to handle both.
  • Never emit JSON or structured data. Goal extraction is a separate call with a different prompt. Mixing the two would break parseJsonResponse.
  • Drive the conversation across four dimensions — reading goal, desired depth, focus areas, output style — in whatever order makes sense given prior preferences. 
// Example model reply that the UI knows how to render:
Got it. How much detail do you want?

>> Brief overview
>> Moderate detail
>> Full deep-dive

On restore (re-opening the panel), renderMessages() still strips the >>  lines but does not recreate the chip buttons — the user has already moved past that turn.

4 · Goal extraction

Turning a transcript into a structured goal

Clicking Set Reading Goal fires chatbot-set-reading-goal. This is a one-shot, separate LLM call — not a continuation of the chat.

Inputs

  • The full session transcript, flattened to role: content per line.
  • The same recent-preferences summary used by the chat prompt.
  • src/prompts/goal-extraction.md with {preferences} substituted.

Output schema

{
  "readingGoal": "single clear sentence",
  "desiredDepth": "brief | moderate | detailed",
  "focusAreas": ["topic1", "topic2"],
  "outputStyle": "summary | explanation
                | bullet-points | conversational",
  "additionalInstructions": "...",
  "language": "en"
}

parseJsonResponse() strips any ```json fences before JSON.parse, so a model that hedges with markdown still works. On success the goal is:

  • Inserted as a new row in the userPreferences store (timestamped, with userId and sessionId).
  • Stamped on the session as extractedGoal for later inspection.
  • Logged under category: 'reading-goal' in researchLogs.
  • Returned to the UI, which renders the preview card.

Apply Goal in the preview card writes the object to chrome.storage.sync.readingGoal — that key is what the renarration pipeline actually reads when adapting page content. Until Apply is clicked, nothing downstream of the chat sees the goal.

6 · Legacy

Persona handlers still in the file

src/handlers/chatbot.js exports two more handlers that the current UI does not surface: chatbot-generate-persona (runs the persona-extraction prompt over the transcript) and chatbot-apply-persona (saves the result into chrome.storage.sync.personas under a generated slug + id). They predate the reading-goal flow and remain wired up in case anything calls them — do not assume the chat itself triggers them.

7 · Gotchas

Things worth knowing before editing

  • Two role spaces. Session messages are stored as user/model; the LLM dispatch maps to user/assistant. Keep the boundary clean when adding new turns — don't write assistant into the session.
  • System prompt is the chip contract. The >>  prefix convention lives only in the system prompt and the UI parser. If you change one, change both.
  • Vite-baked env. VITE_OPENAI_API_KEY and VITE_OPENAI_TEXT_MODEL are inlined at build time. Rotating the key requires a rebuild — the extension never reads them at runtime.
  • Goal is not auto-applied. Extraction writes to IndexedDB, but the user has to click Apply Goal to populate chrome.storage.sync.readingGoal. The pipeline reads only the sync key.
  • No streaming. The Responses call is synchronous-style; the UI shows a static "Thinking..." bubble until the full reply arrives. If a long timeout fires, callOpenAIText throws and the user gets a generic "Sorry, I encountered an error" bubble.
  • Quick replies disappear on send. Whenever the user sends — whether by typing or by clicking a chip — the chip row is hidden. Restored sessions never show chips at all.
